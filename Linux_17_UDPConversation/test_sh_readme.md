别担心，Shell 脚本其实就是把你在终端里手动敲的命令“打包”进一个文本文件，让它自动运行。对于你这种逻辑性强的数学系毕业生来说，这简直就是小菜一碟。

为了帮你克服重构后的焦虑并快速看到压测效果，我们直接写一个最实用的“多进程发包脚本”。

---

### 1. 压测脚本：`stress_test.sh`

你可以直接在你的代码目录下新建一个文本文件，命名为 `stress_test.sh`，内容如下：

```bash
#!/bin/bash

# 1. 设置启动的客户端数量（比如先来 20 个）
CLIENT_COUNT=20

echo "开始启动 $CLIENT_COUNT 个客户端进行压测..."

for i in $(seq 1 $CLIENT_COUNT)
do
    # 2. 模拟用户交互逻辑
    # 利用括号 ( ) 将命令组合，通过 | 传给 ChatClient
    # & 符号让这个客户端在后台运行，不会阻塞循环
    (
        echo "User_$i"             # 第一步：注册用户名
        sleep 0.5                  # 稍微停顿，确保服务器处理完注册
        echo "Server 你好，我是 $i" # 第二步：按照 <用户名> <内容> 格式发消息
        sleep 1                    # 维持连接 1 秒
        echo "exit"                # 第三步：退出
    ) | ./ChatClient > /dev/null 2>&1 & 

    echo "客户端 User_$i 已启动"
done

# 3. 等待所有后台进程结束
wait
echo "压测任务执行完毕！"
```

---

### 2. 脚本关键语法拆解

为了让你理解得更透彻，这里有几个你必须掌握的“知识点”：

* **`#!/bin/bash`**：这叫 Shebang，告诉系统：“嘿，请用 `/bin/bash` 来解释执行这个文件。”
* **`for i in $(seq 1 $CLIENT_COUNT)`**：标准的循环语句，`seq` 会生成从 1 到 20 的数字序列。
* **`&` (Backgrounding)**：这是压测的核心。如果没有它，脚本会等第一个客户端跑完才启动第二个；有了它，20 个客户端会**同时**拔地而起。
* **`|` (Pipe)**：把你 `echo` 出来的字符串“喂”给 `ChatClient` 的标准输入（即 `cin`）。
* **`> /dev/null 2>&1`**：这是“静音插件”。它把客户端输出的各种提示信息全部扔进黑洞，防止把你现在的终端刷屏，让你能看清服务器这边的动静。

### 3. 如何运行它？

作为一名嵌入式开发者，你肯定知道 Linux 下文件是有权限的。

1.  **赋予执行权限**：
    ```bash
    chmod +x stress_test.sh
    ```
2.  **执行压测**：
    ```bash
    ./stress_test.sh
    ```

### 4. 师姐的“实验”指导

由于你现在开启了守护进程模式并将输出重定向到了日志文件，压测时你可以开三个窗口：

* **窗口 A**：运行 `./ChatServer`。
* **窗口 B**：运行 `tail -f log/server_running.log`（监控日志）。
* **窗口 C**：运行 `./stress_test.sh`。

当你运行脚本时，盯着**窗口 B**。如果你在日志里看到类似 `Capacity enlarged!` 的字样（如果你按我说的在 `enlarge_capacity` 里加了打印），那就说明你的 `BufferPool` 真的在像心脏一样跳动、扩张了！
